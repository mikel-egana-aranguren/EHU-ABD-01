MongoDB

BBDD documentales

La unidad básica de almacenamiento es el documento completo, normalmente JSON o BSON

Los documentos se agrupan en colecciones

El esquema es flexible: dos documentos de la misma colección pueden tener atributos distintos

MongoDB es la BBDD documental más popular

Idea principal

En relacional modelamos tablas y relaciones

En documental modelamos objetos completos que la aplicación necesita leer y escribir

Se priorizan consultas simples y rápidas sobre estructuras cercanas al dominio

Esto reduce joins, aunque puede introducir redundancia

Documento de ejemplo

{
  "_id": 101,
  "titulo": "El nombre de la rosa",
  "autores": ["Umberto Eco"],
  "anio": 1980,
  "genero": "novela historica",
  "stock": 12,
  "editorial": {
    "nombre": "Lumen",
    "pais": "ES"
  }
}

Un mismo documento puede mezclar valores simples, listas y subdocumentos

Ventajas

Modelo natural para datos semi-estructurados

Evolución rápida del esquema cuando cambia la aplicación

Buena escalabilidad horizontal y replicación

Muy útil para catálogos, eventos, perfiles, CMS y datos JSON de APIs

Costes y límites

La flexibilidad no elimina la necesidad de diseñar bien el modelo

Puede haber duplicación de datos

Las transacciones multi-documento existen, pero no son el caso ideal de uso

Si el problema depende de joins complejos y consistencia estricta, relacional puede encajar mejor

Embebido vs referencia

Embebido: guardar juntos datos que se leen juntos

Referencia: separar datos grandes, compartidos o con ciclo de vida propio

Regla práctica: modelar según las consultas más frecuentes

En MongoDB el diseño empieza por el patrón de acceso, no por la normalización

Ejemplo embebido

{
  "_id": 9001,
  "cliente": "Ane",
  "fecha": "2026-03-11",
  "lineas": [
    { "producto": "SSD 1TB", "unidades": 1, "precio": 89 },
    { "producto": "RAM 32GB", "unidades": 2, "precio": 95 }
  ],
  "total": 279
}

El pedido se recupera completo con una sola lectura

CRUD básico en MongoDB

use biblioteca

db.libros.insertOne({
  titulo: "Dune",
  autores: ["Frank Herbert"],
  anio: 1965,
  stock: 8
})

db.libros.find({ stock: { $gt: 0 } })

Colección: libros. Base de datos: biblioteca

Consultas frecuentes

db.libros.find(
  { anio: { $gte: 2000 }, genero: "ciencia ficcion" },
  { titulo: 1, autores: 1, _id: 0 }
)

db.libros.find({ "editorial.pais": "ES" })

db.libros.find({ autores: "Umberto Eco" })

Podemos filtrar por campos simples, campos anidados y elementos de arrays

Actualización y borrado

db.libros.updateOne(
  { titulo: "Dune" },
  {
    $set: { stock: 7 },
    $push: { autores: "Brian Herbert" }
  }
)

db.libros.deleteMany({ stock: 0 })

Las operaciones usan operadores como $set, $push, $inc o $unset

Agregación

db.pedidos.aggregate([
  { $unwind: "$lineas" },
  {
    $group: {
      _id: "$lineas.producto",
      unidadesTotales: { $sum: "$lineas.unidades" }
    }
  },
  { $sort: { unidadesTotales: -1 } }
])

La pipeline de agregación permite transformar y resumir colecciones

Índices

Sin índices, MongoDB debe recorrer muchos documentos

Los índices aceleran consulta y ordenación, pero ocupan espacio y encarecen escrituras

db.libros.createIndex({ titulo: 1 })
db.libros.createIndex({ genero: 1, anio: -1 })

Hay que indexar según consultas reales, no por intuición

Cuándo usar MongoDB

Catálogos de productos con atributos variables

Perfiles de usuario y preferencias

Eventos, logs, contenidos JSON y prototipos rápidos

No es la mejor opción si la integridad entre muchas entidades es el requisito central
