Introducción

  • En ciertos entornos, las BBDD SQL pueden ser demasiado rígidas.
  • Ejemplos:
    • Almacenar correos electrónicos o mensajes de texto.
      • La longitud del texto puede variar mucho.
      • Los mensajes pueden tener ficheros adjuntos.
    • Gestionar mensajes de log de servidores
      • Cada mensaje de log puede contener campos diferentes
      • La longitud de cada campo puede variar

Introducción

  • Los sistemas NoSQL (Not Only SQL) ofrecen una alternativa a las BBDD SQL.
  • Características relevantes:
    • No requieren esquema
      • No es necesario definir explícitamente el formato de los datos
    • Facilidad para escalabilidad horizontal
      • Preparados para crecer añadiendo más nodos/VMs
    • Particionado horizontal (sharding)
      • Los datos se distribuyen entre los nodos disponibles
      • Favorece el acceso concurrente

Esquemas

  • Los esquemas de validación tienen 2 opciones que controlan su comportamiento:
    • validationLevel
      • Controla cómo de estricta es la validación. 2 opciones:
        • strict (valor por defecto): Se comprueba toda inserción y modificación.
        • moderate: Las modificaciones a documentos ya existentes no se comprueban.
    • validationAction
      • Indica qué hacer cuando un documento no cumple el esquema. 2 opciones:
        • error (valor por defecto): Se emite un error y se impide la inserción.
        • warn: Se escribe un aviso en el log de MongoDB y se permite la inserción.

Índices

  • Al igual que en los sistemas relacionales, en MongoDB se pueden añadir índices para agilizar consultas.
  • Se definen para uno o varios campos de una colección.
  • Tienen las mismas desventajas que en los sistemas relacionales:
    • Ocupan espacio en disco
    • Hacen que las operaciones de actualización se ralenticen

Índices

  • Añadir índice al campo de una colección:
  • donde:
    • campo: Campo al que queremos añadir el índice
    • ORDEN: 1 para que el índice sea ascendente, -1 para que sea descendente.
  • En general, el orden del índice no afecta en exceso.

Copias de seguridad

  • Realizar copias de seguridad periódicas es una tarea fundamental de administración.
  • Hay 2 formas principales en MongoDB:
    • Utilizando sus propias plataformas en la nube (p.e. Atlas).
      • Requiere contratar el servicio.
    • De forma manual
      • Lo veremos en este tema.

Copias de seguridad

  • mongodump realiza una copia de una BD en ficheros BSON.
  • donde:
    • <usuario>: Usuario para autenticarse a MongoDB y realizar la copia
    • <BD-aut>: BD vinculada al usuario para autenticación
    • <BD>: BD que se quiere copiar. Si este parámetro se omite, se copian todas.
  • Se crea una carpeta dump con los ficheros BSON.
  • Es necesario que el usuario tenga permisos para leer las BDs que se quieran copiar.

$ mongodump -u <usuario> --authenticationDatabase <BD-aut> -d <BD>

Copias de seguridad

  • mongorestore restaura una copia hecha con mongodump.
  • donde:
    • <usuario>: Usuario para autenticarse a MongoDB y restaurar la copia
    • <BD-aut>: BD vinculada al usuario para autenticación
    • <BD>: BD en la que restaurar la copia. Opcional.
    • <carpeta-dump>: Ruta a la carpeta con la copia hecha por mongodump.
  • Es necesario que el usuario tenga permisos para escribir en las BDs que se quieren restaurar.

$ mongorestore -u <usuario> --authenticationDatabase <BD-aut>

-d <BD> <carpeta-dump>

Copias de seguridad

  • mongodump y mongorestore necesitan que MongoDB esté en marcha para realizar/restaurar una copia de seguridad.
    • Su uso tiene un impacto en el rendimiento de MongoDB.
  • Una alternativa es copiar las carpetas que contienen los datos con otras herramientas (p.e. rsync)
    • Desventaja: las copias ocupan más que con mongodump.
    • Más información:
      • https://www.mongodb.com/docs/manual/core/backups/#back-up-by- copying-underlying-data-files

  • Hasta ahora hemos trabajado con una única instancia de MongoDB.

Sharding

  • Los documentos de las colecciones se distribuyen entre Shards en base a una “Shard key”.
    • Es uno de los campos de los documentos, elegida por el desarrollador y/o administrador.
    • En el ejemplo de la diapositiva 20, sería el título del libro.
  • Debe elegirse con cuidado, ya que afecta al rendimiento del sistema.
    • Limita la concurrencia.

22

Sharding

  • 2 escenarios posibles en una instalación distribuida
    • Consulta que no hace uso de Shard Key
      • mongos debe enviar la consulta a todos los Shards.
      • Mayor saturación de la red.

Imagen de: https://www.mongodb.com/docs/manual/core/sharded-cluster-query-router/

Sharding

  • 2 escenarios posibles en una instalación distribuida
    • Consulta que hace uso de Shard Key
      • mongos envía la consulta al Shards (o Shards) que contienen la respuesta.
      • Mejor tiempo de respuesta, menor uso de ancho de banda.

Imagen de: https://www.mongodb.com/docs/manual/core/sharded-cluster-query-router/

Sharding

  • Cuándo considerar una instalación distribuida de MongoDB:
    • Para atender consultas con muy baja latencia.
    • Para gestionar conjuntos de datos muy grandes.
  • Una instalación distribuida provee alta disponibilidad y alto rendimiento pero es compleja de gestionar.

    • https://www.mongodb.com/docs/manual/sharding/
