Labelled Property Graphs

Representación de datos en forma de grafos con etiquetas y propiedades

Usados en bases de datos de grafos, análisis de redes sociales, sistemas de recomendación, etc.

Modelo LPG

  • Los nodos representan entidades del dominio.
  • Las relaciones son dirigidas y tienen un tipo, por ejemplo FRIEND_OF o LIVES_IN.
  • Los nodos pueden tener etiquetas, por ejemplo Person o City.
  • Los nodos y las relaciones pueden almacenar propiedades clave-valor.
  • El modelo esta pensado para recorrer caminos y detectar patrones en el grafo.

RDF frente a LPG

RDF: triples homogéneos

LPG: nodos, aristas y propiedades

Neo4j

  • Es la base de datos property graph mas extendida en docencia e industria.
  • Implementa un motor orientado a recorridos sobre nodos y relaciones.
  • Su lenguaje de consulta es Cypher.
  • Ofrece herramientas graficas como Browser y despliegue gestionado mediante AuraDB.
  • Es una referencia practica para entender el paradigma LPG, aunque no sea un estandar abierto como RDF.

Modelo de datos en Neo4j

Ejemplo de patron tipico en Neo4j:

(alice:Person {name: 'Alice', age: 24})
-[:FRIEND_OF {since: 2023}]->
(bob:Person {name: 'Bob', age: 27})

(alice)-[:LIVES_IN]->
(:City {name: 'Bilbo'})

Las etiquetas clasifican nodos y las propiedades añaden contexto sin tener que crear nuevos nodos para cada dato simple.

Cypher: creación de datos

CREATE (Alice:Person {name: 'Alice', age: 24})
CREATE (Bob:Person {name: 'Bob', age: 27})
CREATE (Carol:Person {name: 'Carol', age: 22})
CREATE (Dave:Person {name: 'Dave', age: 25})
CREATE (Alice)-[:FRIEND_OF]->(Bob)
CREATE (Alice)-[:FRIEND_OF]->(Carol)
CREATE (Bob)-[:FRIEND_OF]->(Dave)
CREATE (Carol)-[:FRIEND_OF]->(Dave)

Cypher describe patrones del grafo de forma declarativa, muy cercana al dibujo mental del dominio.

Cypher: consulta de patrones

MATCH (alice:Person {name: 'Alice'})-[:FRIEND_OF]->(friends)
RETURN friends.name
MATCH (person:Person)-[:FRIEND_OF]->(friend)
RETURN person.name, count(friend) AS total_friends
ORDER BY total_friends DESC

La idea central es buscar subgrafos que encajan con un patron y devolver nodos, relaciones o agregados.

Cypher: caminos

MATCH p = shortestPath(
  (:Person {name: 'Alice'})-[:FRIEND_OF*..5]->(:Person {name: 'Dave'})
)
RETURN p
  • Muy util para redes sociales, fraude, rutas logisticas o dependencias.
  • La consulta se formula como un patron de camino, no como una cascada de joins.

Cuando encaja Neo4j

  • Consultas con muchos saltos entre entidades relacionadas.
  • Recomendacion, deteccion de comunidades, fraude, rutas y conocimiento conectado.
  • Dominios donde el esquema evoluciona y aparecen nuevas relaciones con frecuencia.
  • Casos donde interesa explorar visualmente el grafo y combinar consultas con algoritmos de grafos.

PageRank en Neo4j

MATCH (source:Page)-[:LINKS_TO]->(target:Page)
RETURN source, target

CALL gds.graph.project(
  'webGraph',
  'Page',
  'LINKS_TO'
)

CALL gds.pageRank.stream('webGraph')
YIELD nodeId, score
RETURN gds.util.asNode(nodeId).name AS page, score
ORDER BY score DESC
  • PageRank estima la importancia de un nodo segun los enlaces que recibe desde otros nodos relevantes.
  • Es util en recomendacion, analisis web, influencia en redes sociales y deteccion de entidades centrales.
  • Neo4j lo ofrece dentro de Graph Data Science junto con muchos otros algoritmos de grafos.

Limites de Neo4j

  • No ofrece la interoperabilidad abierta de RDF ni sus estandares W3C.
  • El razonamiento semantico y la integracion web de datos no son su objetivo principal.
  • Conviene distinguir entre analisis de grafos conectados y publicacion/integracion semantica de datos.
  • En la practica: Neo4j para traversal y analitica; RDF para intercambio, semantica e interoperabilidad.

Buenas practicas de modelado

  • Usar etiquetas estables y expresivas, por ejemplo Person, City, Company.
  • Guardar como propiedades solo los atributos simples; convertir en nodos las entidades con vida propia.
  • Nombrar las relaciones como verbos o roles claros: WORKS_AT, LOCATED_IN, FRIEND_OF.
  • Crear indices y restricciones sobre identificadores naturales o tecnicos.

MERGE, restricciones e indices

CREATE CONSTRAINT person_name IF NOT EXISTS
FOR (person:Person) REQUIRE person.name IS UNIQUE;

MERGE (alice:Person {name: 'Alice'})
ON CREATE SET alice.age = 24
ON MATCH SET alice.last_seen = date();

CREATE INDEX city_name IF NOT EXISTS
FOR (city:City) ON (city.name);
  • MERGE evita duplicados cuando queremos asegurar la existencia de un patron.
  • Las restricciones protegen la integridad de identificadores clave.
  • Los indices aceleran busquedas frecuentes por propiedades.
