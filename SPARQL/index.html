GROUP BY

Agrupar resultados por valores de una variable

SELECT ?ciudad (COUNT(?museo) AS ?numMuseos)

WHERE {

  GRAPH <urn:count> {

    ?museo rdf:type dbpedia:museum ;

    gip_prop:kokalekua ?ciudad .

  }

}

GROUP BY ?ciudad

ORDER BY DESC(?numMuseos)

HAVING

Filtrar grupos después de la agregación

SELECT ?ciudad (COUNT(?museo) AS ?numMuseos)

WHERE {

  GRAPH <urn:count> {

    ?museo rdf:type dbpedia:museum ;

    gip_prop:kokalekua ?ciudad .

  }

}

GROUP BY ?ciudad

HAVING (COUNT(?museo) > 1)

ORDER BY DESC(?numMuseos)

BIND

SELECT ?person ?fullName

WHERE {

  GRAPH <urn:name-surname>{

    ?person foaf:firstName ?first ;

    foaf:lastName ?last .

  }

BIND(CONCAT(?first, " ", ?last) AS ?fullName)

}

MINUS

Excluir resultados que coinciden con un patrón: todos los museos menos los museos que tienen más de 300 obras

SELECT ?museo ?obras

WHERE {

  ?museo rdf:type dbpedia:museum .

  MINUS {

    ?museo gip_prop:kopurua ?obras .

    FILTER (?obras > 300)

  }

}

NOT EXISTS

Filtrar basándose en la no existencia de un patrón

EXISTS vs MINUS

EXISTS/NOT EXISTS: filtro, no elimina bindings

MINUS: elimina soluciones completas

NOT EXISTS es más eficiente para validación

MINUS es más útil para exclusión de patrones complejos

Property Paths

^: inverso (ex:parent^ encuentra hijos)

*: cero o más (ex:knows* para red de conocidos)

+: uno o más (ex:subClassOf+ para jerarquía)

!: cualquier relacion menos esa

Property Paths

?: opcional (cero o uno)

|: alternativa (ex:email|ex:phone)

/: secuencia (ex:livesIn/ex:country)

LOAD

Cargar RDF desde una URI a un grafo específico:

LOAD <http://example.org/data.rdf>

INTO GRAPH <http://example.org/myGraph>



# O sin especificar grafo (va al grafo por defecto):

LOAD <http://example.org/data.rdf>

CLEAR

Eliminar todo el contenido de un grafo:

CLEAR GRAPH <http://example.org/myGraph>



# Limpiar el grafo por defecto:

CLEAR DEFAULT



# Limpiar todos los grafos:

CLEAR ALL

CREATE

Crear un nuevo grafo con nombre:

CREATE GRAPH <http://example.org/newGraph>



# Crear solo si no existe:

CREATE SILENT GRAPH <http://example.org/newGraph>

DROP

Eliminar un grafo completamente:

DROP GRAPH <http://example.org/myGraph>



# Eliminar sin errores si no existe:

DROP SILENT GRAPH <http://example.org/myGraph>



# Eliminar todos los grafos:

DROP ALL

COPY

Copiar el contenido de un grafo a otro:

COPY <http://example.org/sourceGraph>

TO <http://example.org/targetGraph>



# Copiar grafo por defecto a uno nombrado:

COPY DEFAULT

TO <http://example.org/backup>

MOVE

Mover el contenido de un grafo a otro (borra el origen):

MOVE <http://example.org/sourceGraph>

TO <http://example.org/targetGraph>



# Mover del grafo por defecto:

MOVE DEFAULT

TO <http://example.org/newLocation>

ADD

Añadir el contenido de un grafo a otro (sin borrar el origen):

ADD <http://example.org/sourceGraph>

TO <http://example.org/targetGraph>



# Añadir al grafo por defecto:

ADD <http://example.org/toMerge>

TO DEFAULT

Graph Management: Operaciones Conjuntas

Ejemplo: Crear, cargar, copiar y limpiar grafos:

# 1. Crear nuevo grafo

CREATE GRAPH <http://example.org/staging>;



# 2. Cargar datos

LOAD <http://example.org/data.rdf>

INTO GRAPH <http://example.org/staging>;



# 3. Copiar al destino

COPY <http://example.org/staging>

TO <http://example.org/production>;



# 4. Limpiar staging

CLEAR GRAPH <http://example.org/staging>

COPY / MOVE / ADD

COPY: copiar un grafo a otro (sobreescribe destino)

MOVE: mover grafo (copia + elimina origen)

ADD: añadir triples al destino (sin sobreescribir)

COPY GRAPH <http://example.org/graph1> 
TO GRAPH <http://example.org/graph2>

SPARQLWrapper - Ejemplo Python

from SPARQLWrapper import SPARQLWrapper, JSON

sparql = SPARQLWrapper("https://query.wikidata.org/sparql")
sparql.setQuery("""
    SELECT ?city ?cityLabel WHERE {
        ?city wdt:P31 wd:Q515 .
        SERVICE wikibase:label { 
            bd:serviceParam wikibase:language "en" 
        }
    } LIMIT 10
""")
sparql.setReturnFormat(JSON)
results = sparql.query().convert()

SPARQL 1.1 Service Description

Metadatos sobre el endpoint SPARQL

Descubrir capacidades del servidor

SELECT * WHERE {
    <http://example.org/sparql> ?p ?o
}

Información: datasets, grafos, extensiones, límites

Consultas Federadas (SERVICE)

Permiten consultar múltiples endpoints SPARQL en una sola query

Combinar datos de fuentes distribuidas usando la palabra clave SERVICE

SELECT ?persona ?interes
WHERE {
  ?persona foaf:name "Alice" .
  SERVICE <http://endpoint-remoto.example/sparql> {
    ?persona foaf:interest ?interes .
  }
}

Service Description: Propósito

Proporciona metadatos sobre endpoints SPARQL para ayudar a los clientes:

  • Descubrir capacidades: ¿Qué funcionalidades soporta? (agregaciones, property paths, negación, etc.)
  • Encontrar datasets disponibles: ¿Qué grafos con nombre y grafo por defecto están disponibles?
  • Determinar formatos soportados: JSON, XML, CSV, etc.
  • Ver regímenes de inferencia: RDFS, OWL, etc.

Relación: Service Description + Federación

Service Description permite la construcción inteligente de consultas federadas

Antes de enviar queries federadas mediante SERVICE, se puede:

  • Consultar la descripción para verificar si el endpoint remoto soporta las funcionalidades necesarias
  • Evitar errores en tiempo de ejecución descubriendo limitaciones previamente
  • Optimizar consultas según los datasets disponibles en cada endpoint
  • Descubrir y seleccionar dinámicamente endpoints apropiados para la federación

Service Description + Federación: Ejemplo

Descubrir capacidades antes de federar:

# 1. Consultar Service Description
SELECT ?feature WHERE {
  <http://ejemplo.org/sparql> sd:feature ?feature
}

# 2. Si soporta agregaciones, ejecutar consulta federada
SELECT ?ciudad (COUNT(?museo) AS ?total)
WHERE {
  ?ciudad a ex:Ciudad .
  SERVICE <http://ejemplo.org/sparql> {
    ?museo ex:ubicadoEn ?ciudad
  }
}
GROUP BY ?ciudad

Ventajas de combinar ambas

  • Robustez: Verificar compatibilidad antes de ejecutar
  • Eficiencia: Seleccionar el mejor endpoint según capacidades
  • Adaptabilidad: Ajustar consultas según los recursos disponibles
  • Descubrimiento dinámico: Encontrar automáticamente endpoints adecuados

En resumen: Service Description responde "¿qué puedo hacer?", haciendo las consultas federadas más fiables y eficientes

SPARQL vs SQL

Similitudes: SELECT, WHERE, FILTER (WHERE), ORDER BY, LIMIT

Diferencias:

  • SPARQL: patrón de grafos; SQL: tablas relacionales
  • SPARQL: datos distribuidos, federación; SQL: local
  • SPARQL: esquema flexible; SQL: esquema rígido
  • SPARQL: URIs globales; SQL: IDs locales
