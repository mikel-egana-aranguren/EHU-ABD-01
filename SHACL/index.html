SHACL

SHApes Constraint Language

Define la forma que un grafo RDF debería tener: que relaciones, a que nodos, con que valores, etc.

Se validan datos RDF contra esa forma, y se genera un informe que dice si los datos cumplen la forma o no (ValidationReport, en RDF)

SHACL

SHACL es un recomendación (Estándar) del W3C: (W3C)

El vocabulario SHACL, definido en RDF, contiene todo el lenguaje SHACL

El mismo lenguaje RDF para definir datos (RDF) esquema (SHACL) e informes de resultados (ValidationReport) (NoSQL!RDF!)

SHACL vs OWL

Confusión muy común: OWL como lenguaje de esquema

OWL es un lenguaje de inferencia, no de esquema (*): por ejemplo, no dice cuántas relaciones tiene que tener una instancia, si no que si nos encontramos una instancia con esas relaciones, se infiere que pertenece a cierta clase

(*) Se puede conseguir "cerrar" el mundo en OWL, pero con ciertos axiomas y no por defecto

SHACL vs OWL

Confusión muy común: OWL como lenguaje de esquema

SHACL sí es un lenguaje de esquema: por ejemplo, nos dice cuántas relaciones tiene que tener una instancia, y si esa instancia no las tiene, es inválida

Sin embargo, SHACL está evolucionando hacia sustituir a OWL en ciertos contextos

Shape SHACL

SHACL: Componentes Core

sh:NodeShape: define restricciones sobre nodos del grafo

sh:PropertyShape: define restricciones sobre propiedades

sh:targetClass, sh:targetNode, sh:targetSubjectsOf, sh:targetObjectsOf: qué nodos validar

sh:path: qué propiedad validar

SHACL: Restricciones de cardinalidad

sh:minCount: número mínimo de valores

sh:maxCount: número máximo de valores

ex:PersonShape a sh:NodeShape ;
    sh:targetClass ex:Person ;
    sh:property [
        sh:path ex:name ;
        sh:minCount 1 ;
        sh:maxCount 1
    ] .

SHACL: Restricciones de tipo

sh:class: el valor debe ser instancia de una clase

sh:datatype: el valor debe tener un datatype específico

sh:nodeKind: tipo de nodo (IRI, Literal, BlankNode)

sh:property [
    sh:path ex:birthDate ;
    sh:datatype xsd:date ;
    sh:maxCount 1
] .

SHACL: Restricciones de valor

sh:in: el valor debe estar en una lista específica

sh:hasValue: debe existir un valor específico

sh:minExclusive, sh:maxExclusive: rangos numéricos

sh:minInclusive, sh:maxInclusive: rangos numéricos inclusivos

SHACL: Restricciones de cadenas

sh:minLength, sh:maxLength: longitud de strings

sh:pattern: expresión regular que debe cumplir

sh:languageIn: idiomas permitidos para literales

sh:uniqueLang: cada idioma solo puede aparecer una vez

SHACL: Restricciones de cadenas

sh:property [
    sh:path rdfs:label ;
    sh:uniqueLang true
] .

SHACL: Restricciones lógicas

sh:not: negación de una shape

sh:and: conjunción (todas las shapes deben cumplirse)

sh:or: disyunción (al menos una shape debe cumplirse)

sh:xone: disyunción exclusiva (exactamente una debe cumplirse)

SHACL: Restricciones de forma

sh:node: el valor debe cumplir otra shape

sh:property: define restricciones anidadas

sh:qualifiedValueShape + sh:qualifiedMinCount/MaxCount: cuántos valores deben cumplir una shape

Permite validaciones complejas y composición de shapes

SHACL: Closed Shapes

sh:closed true: solo se permiten las propiedades declaradas

sh:ignoredProperties: lista de propiedades que se ignoran

ex:PersonShape a sh:NodeShape ;
    sh:targetClass ex:Person ;
    sh:closed true ;
    sh:ignoredProperties ( rdf:type ) ;
    sh:property [
        sh:path ex:name
    ] .

SHACL: Severity Levels

sh:severity: nivel de gravedad de una violación

sh:Violation (por defecto): error crítico

sh:Warning: advertencia

sh:Info: información

Permite diferenciar validaciones obligatorias de recomendaciones

SHACL: Mensajes personalizados

sh:message: mensaje descriptivo para violaciones

Puede ser multilingüe usando language tags

sh:property [
    sh:path ex:age ;
    sh:datatype xsd:integer ;
    sh:minInclusive 0 ;
    sh:message "La edad debe ser un entero no negativo"@es ;
    sh:message "Age must be a non-negative integer"@en
] .

SHACL: Property Paths

sh:path soporta caminos complejos:

Secuencia: ( ex:parent ex:age ) - padre.edad

Alternativa: [ sh:alternativePath ( ex:email ex:phone ) ]

SHACL: Property Paths

Inverso: [ sh:inversePath ex:knows ]

Zero o más: [ sh:zeroOrMorePath ex:parent ]

Uno o más: [ sh:oneOrMorePath ex:subClassOf ]

SHACL-SPARQL

Extensión avanzada: validaciones con queries SPARQL

sh:sparql permite restricciones arbitrariamente complejas

Más expresivo que SHACL Core pero menos eficiente

Útil para validaciones que requieren razonamiento complejo

SHACL-SPARQL: Ejemplo

ex:UniqueEmailShape a sh:NodeShape ;
    sh:targetClass ex:Person ;
    sh:sparql [
        sh:message "Email must be unique" ;
        sh:select """
            SELECT $this ?email WHERE {
                $this ex:email ?email .
                ?other ex:email ?email .
                FILTER ($this != ?other)
            }
        """
    ] .

SHACL: Validation Report

Resultado de validación en RDF (NoSQL!RDF!)

sh:ValidationReport con sh:conforms (true/false)

SHACL: Validation Report

sh:ValidationResult para cada violación:

  • sh:focusNode: nodo que viola la restricción
  • sh:resultPath: propiedad violada
  • sh:value: valor problemático
  • sh:resultMessage: descripción del error

SHACL 1.2

Working Draft del W3C (marzo 2026), una evolución significativa respecto a SHACL 1.0

Nuevo alcance ampliado: "SHACL se puede usar para validación, inferencia, modelado de dominios, generación de interfaces de usuario, generación de código e integración de datos"

SHACL 1.2

  • SHACL 1.2 Core: núcleo del lenguaje
  • SHACL 1.2 SPARQL Extensions: definir restricciones basadas en SPARQL
  • SHACL 1.2 Rules (SRL): inferencia basada en reglas
  • SHACL 1.2 UI: generación de interfaces de usuario
  • SHACL 1.2 Profiling: crear perfiles que representan a grafos

SHACL 1.2 Core: Novedades principales

Nuevos mecanismos de target: sh:targetWhere, sh:shape

Nuevas restricciones: sh:someValue, sh:singleLine, restricciones de listas

sh:ShapeClass: una clase que es al mismo tiempo una shape

Soporte para RDF 1.2: sh:reifierShape

Propiedades no-validantes para IA: sh:intent, sh:agentInstruction

Core: sh:ShapeClass

Una clase que también actúa como shape, simplificando la definición

Todas las instancias de la clase se validan automáticamente contra la shape

Equivale a definir sh:NodeShape + sh:targetClass por separado

Core: sh:ShapeClass

ex:Person a sh:ShapeClass, rdfs:Class ;
    sh:property [
        sh:path ex:name ;
        sh:minCount 1 ;
        sh:datatype xsd:string
    ] .

Core: sh:targetWhere

Nuevo mecanismo de target basado en condiciones de la shape

Los nodos del grafo de datos que cumplan las condiciones serán focus nodes

Core: sh:targetWhere

ex:HighScoreShape a sh:NodeShape ;
    sh:targetWhere [
            sh:property [
                sh:path ex:score ;
                sh:minInclusive 90
            ]
        ] ;
            sh:property [
            sh:path ex:award ;
            sh:minCount 1
    ] .

Core: sh:shape como target implícito

sh:shape: marca una shape como objetivo implícito, validando automáticamente cualquier nodo que se encuentre en el grafo y que sea candidato

A diferencia de sh:targetClass, no requiere que los nodos sean explícitamente instancias de una clase

Core: sh:shape como target implícito

ex:PersonShape a sh:NodeShape ;
    sh:shape true ;
    sh:property [
        sh:path ex:name ;
        sh:minCount 1
    ] .

Core: Node Expressions

Permiten computar dinámicamente valores de propiedades

sh:values: asigna valores derivados a una propiedad

Funciones: sh:concat, sh:limit, sh:offset, sh:orderBy, sh:count, sh:sum...

Core: Node Expressions

ex:Person-fullName a sh:PropertyShape ;
    sh:path ex:fullName ;
    sh:values [
        sh:concat (
            [ sh:path ex:firstName ]
            " "
            [ sh:path ex:lastName ]
        )
    ] .

Core: Nuevos niveles de severidad

Además de sh:Violation, sh:Warning, sh:Info, SHACL 1.2 añade:

  • sh:Trace: para depuración durante desarrollo
  • sh:Debug: información de diagnóstico

Core: Propiedades para Agentes IA

Nuevas propiedades no-validantes para integración con agentes inteligentes:

  • sh:intent: describe la intención de la shape (para humanos y agentes)
  • sh:agentInstruction: instrucciones específicas para agentes IA/LLM

Core: sh:singleLine y sh:conformanceDisallows

sh:singleLine: indica si un valor literal debe ser una sola línea (sin saltos de línea). Usado por interfaces de usuario para elegir el widget adecuado

sh:conformanceDisallows: permite definir qué impide la conformancia, ampliando el modelo de validación

SHACL 1.2 Rules

Nueva especificación para inferencia basada en reglas

Introduce el Shape Rules Language (SRL): un lenguaje dedicado con extensión de fichero ".srl"

Las reglas se pueden adjuntar a shapes para generar nuevos triples

Soporta recursión y negación como fallo (negation as failure)

Rules: Sintaxis SRL

Las reglas usan sintaxis RULE/WHERE:

PREFIX ex: <http://example.com/ns#>

RULE {
    ?person ex:fullName ?fullName .
}
WHERE {
    ?person ex:firstName ?first .
    ?person ex:lastName ?last .
    BIND(CONCAT(?first, " ", ?last) AS ?fullName)
}

La cláusula WHERE define condiciones; la cláusula RULE define los triples a generar

Rules: Negación y recursión

Negación como fallo: se pueden definir reglas que aplican cuando algo NO existe

RULE {
    ?person ex:status "Sin email" .
}
WHERE {
    ?person a ex:Person .
    NOT EXISTS { ?person ex:email ?email }
}

Recursión: las reglas pueden generar triples que activan otras reglas, con estratificación para garantizar resultados consistentes

Rules: Adjuntar reglas a Shapes

Las reglas se pueden vincular a shapes específicas

Esto permite que las reglas solo apliquen a los focus nodes de la shape

ex:PersonShape a sh:NodeShape ;
    sh:targetClass ex:Person ;
    sh:rule [
        a sh:SRLRule ;
        sh:construct """
            RULE { $this ex:fullName ?fn }
            WHERE {
                $this ex:firstName ?f .
                $this ex:lastName ?l .
                BIND(CONCAT(?f," ",?l) AS ?fn)
            }
        """
    ] .

Rules: Estratificación

Las reglas se evalúan en capas de estratificación para resolver dependencias y negación:

  1. Se analiza el grafo de dependencias entre reglas
  2. Las reglas que dependen de negaciones de otras se colocan en capas superiores
  3. Cada capa se evalúa completamente antes de pasar a la siguiente

Garantiza un modelo semántico único y consistente, similar a Datalog estratificado

Rules: Sintaxis RDF alternativa

Las reglas SRL también se pueden expresar directamente en RDF (sintaxis RDF Rules)

Útil cuando se quieren almacenar las reglas dentro del mismo grafo RDF

ex:FullNameRule a sh:SRLRule ;
    sh:condition [
        sh:property [
            sh:path ex:firstName ;
            sh:minCount 1
        ]
    ] ;
    sh:rule [
        sh:subject sh:this ;
        sh:predicate ex:fullName ;
        sh:object [ sh:concat (
            [ sh:path ex:firstName ] " "
            [ sh:path ex:lastName ] ) ]
    ] .

SHACL 1.2 SPARQL Extensions

Define extensiones de SHACL basadas en SPARQL 1.2

Tres pilares principales:

  • Restricciones basadas en SPARQL (sh:sparql)
  • Componentes de restricción reutilizables (sh:ConstraintComponent)
  • Expresiones de nodo SPARQL (sh:SelectExpression, sh:SPARQLExprExpression)

Más expresivo que SHACL Core pero menos eficiente

SPARQL: Restricciones con sh:sparql

Permiten restricciones arbitrariamente complejas usando SELECT queries

ex:LanguageShape a sh:NodeShape ;
    sh:targetClass ex:Country ;
    sh:sparql [
        sh:message "Debe tener etiqueta en alemán" ;
        sh:prefixes ex: ;
        sh:select """
            SELECT $this ?value WHERE {
                $this ex:germanLabel ?value .
                FILTER (!langMatches(lang(?value), "de"))
            }
        """
    ] .

La variable $this se pre-asocia al focus node actual

SPARQL: Componentes de restricción

Permiten abstraer y reutilizar lógica de validación SPARQL

Se declaran con sh:ConstraintComponent + sh:parameter + validador

ex:LangComponent a sh:ConstraintComponent ;
    sh:parameter [ sh:path ex:lang ] ;
    sh:validator [
        a sh:SPARQLAskValidator ;
        sh:message "Idioma incorrecto: {$lang}" ;
        sh:ask """ASK {
            FILTER (isLiteral($value) &&
              langMatches(lang($value), $lang))
        }"""
    ] .

SPARQL: Validadores ASK vs SELECT

Dos tipos de validadores SPARQL:

  • ASK (sh:SPARQLAskValidator): devuelve true si el valor cumple la restricción — lógica directa
  • SELECT (sh:SPARQLSelectValidator): devuelve los valores que NO cumplen — lógica inversa

Selección de validador: sh:nodeValidator (para NodeShape), sh:propertyValidator (para PropertyShape), sh:validator (para ambos, solo ASK)

SPARQL: Annotation Properties

Novedad en SHACL 1.2: permite inyectar propiedades adicionales en los resultados de validación

ex:AuditShape a sh:NodeShape ;
    sh:sparql [
        sh:resultAnnotation [
            sh:annotationProperty ex:timestamp ;
            sh:annotationVarName "time"
        ] ;
        sh:select """
            SELECT $this ?message ?time WHERE {
                BIND(NOW() AS ?time)
                BIND("Auditado" AS ?message)
            }
        """
    ] .

El resultado de validación incluirá ex:timestamp con el valor de ?time

SPARQL: Node Expressions SPARQL

Dos nuevas funciones de expresión de nodo basadas en SPARQL:

sh:SelectExpression: computa valores usando un SELECT completo

ex:Person-fullName a sh:PropertyShape ;
    sh:path ex:fullName ;
    sh:values [
        sh:select """
            SELECT ?fullName WHERE {
                $this ex:firstName ?f .
                $this ex:lastName ?l .
                BIND(CONCAT(?f," ",?l) AS ?fullName)
            }
        """
    ] .

SPARQL: sh:SPARQLExprExpression

sh:sparqlExpr: expresión SPARQL compacta para computar un valor

ex:Resource-uriLength a sh:PropertyShape ;
    sh:path ex:uriLength ;
    sh:values [
        sh:sparqlExpr "STRLEN(STR($this))"
    ] ;
    sh:datatype xsd:integer .

Más conciso que un SELECT completo para expresiones simples

Se ha eliminado el soporte para las variables pre-asociadas opcionales $shapesGraph y $currentShape

SPARQL: Prefijos para queries

Se pueden declarar prefijos reutilizables en el shapes graph:

ex:
    sh:declare [
        sh:prefix "ex" ;
        sh:namespace
            "http://example.com/ns#"^^xsd:anyURI
    ] ;
    sh:declare [
        sh:prefix "schema" ;
        sh:namespace
            "http://schema.org/"^^xsd:anyURI
    ] .

Las restricciones SPARQL referencian estos prefijos con sh:prefixes

SHACL 1.2 UI (User Interfaces)

Nueva especificación para generación de interfaces de usuario a partir de shapes SHACL

Define el concepto de SHACL Renderer: software que procesa shapes y datos para generar UI dinámicamente

Nuevo namespace: shui: (http://www.w3.org/ns/shacl-ui#)

Dos modos: vista (visualización) y edición (modificación de datos)

UI: Widgets (Editores y Visores)

Los widgets se dividen en:

  • Editores (shui:Editor): para modificar datos — TextFieldEditor, DatePickerEditor, BooleanSelectEditor, AutoCompleteEditor, EnumSelectEditor...
  • Visores (shui:Viewer): para mostrar datos — LabelViewer, LiteralViewer, HTMLViewer, ImageViewer, URIViewer...

Se seleccionan manualmente (shui:editor, shui:viewer) o automáticamente mediante un sistema de puntuación

UI: Ejemplo de editores

ex:Person-married a sh:PropertyShape ;
    sh:path ex:married ;
    sh:datatype xsd:boolean .
# → BooleanSelectEditor (score 10)

ex:Person-dateOfBirth a sh:PropertyShape ;
    sh:path ex:dateOfBirth ;
    sh:datatype xsd:date .
# → DatePickerEditor (score 10)

ex:Address-region a sh:PropertyShape ;
    sh:path schema:addressRegion ;
    sh:in ("ACT" "NSW" "QLD" "SA" "VIC") .
# → EnumSelectEditor (score 10)

UI: Property Paths en modo vista y edición

Modo vista: MUST soportar predicate paths e inverse paths; SHOULD soportar paths complejos (secuencia, alternativa, etc.)

Modo edición: MUST soportar predicate e inverse paths; SHOULD soportar alternative paths

ex:PersonShape a sh:NodeShape ;
    sh:property [
        sh:path foaf:name ;
        sh:name "Nombre"
    ] ;
    sh:property [
        sh:path [ sh:inversePath ex:member ] ;
        sh:name "Departamento"
    ] .

UI: Property Roles

Anotaciones para indicar el rol de una propiedad en la interfaz

shui:propertyRole: indica cómo debe tratarse una propiedad (ej. como etiqueta de visualización)

ex:Label a sh:PropertyShape ;
    sh:path skos:prefLabel ;
    shui:propertyRole shui:LabelRole .

Con prioridad cualificada (sh:order) y soporte para anotaciones de triples RDF 1.2:

ex:PrefLabel a sh:PropertyShape ;
    sh:path skos:prefLabel ;
    shui:propertyRole shui:LabelRole
        {| sh:order 0 |} .

UI: DetailsEditor y formularios anidados

shui:DetailsEditor: permite editar nodos complejos como formularios anidados

ex:Product-weight a sh:PropertyShape ;
    sh:path ex:weight ;
    shui:editor shui:DetailsEditor ;
    sh:maxCount 1 ;
    sh:node ex:ValueWithWeight ;
    sh:nodeKind sh:BlankNode .

ex:ValueWithWeight a sh:NodeShape ;
    sh:property [
        sh:path ex:numericValue ;
        sh:datatype xsd:decimal ;
        sh:minCount 1
    ] ;
    sh:property [
        sh:path ex:unit ;
        sh:class qudt:Unit ;
        sh:minCount 1
    ] .

UI: ValueTableViewer

shui:ValueTableViewer: muestra múltiples valores como una tabla HTML

Es un Multi Viewer: toma control sobre todos los valores de una propiedad

Las columnas se derivan de la node shape referenciada con sh:node

El orden de columnas se controla con sh:order

ex:Concept-narrower a sh:PropertyShape ;
    sh:path [ sh:inversePath skos:broader ] ;
    sh:name "narrower (tabla)" ;
    shui:viewer shui:ValueTableViewer ;
    sh:node ex:ConceptTableShape .

Resumen: Cambios clave en SHACL 1.2

  • Core: sh:ShapeClass, sh:targetWhere, node expressions, restricciones de listas, soporte RDF 1.2, propiedades para agentes IA
  • Rules: Shape Rules Language (SRL), recursión, negación, estratificación
  • SPARQL: annotation properties, SelectExpression, SPARQLExprExpression
  • UI: SHACL Renderer, widgets con puntuación, property roles, formularios anidados
